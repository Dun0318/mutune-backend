# MuTune Backend

Frontend Repository:
https://github.com/Dun0318/mutune-frontend

MuTune은 음악인을 위한 웹 기반 음악 편집 SaaS 서비스입니다.

현재는 악보(PDF) 처리 기능을 중심으로 개발을 진행하고 있으며,  
웹 환경에서 악보를 변환하고 정렬하여 재사용 가능한 구조로 만드는 것을 1차 목표로 하고 있습니다.

이후 단계적으로 MR(반주 파일) 키/템포 편집 기능과  
자동 믹싱 기능까지 확장하는 것을 목표로 설계된 프로젝트입니다.

장기적으로는 단순 편집 도구를 넘어  
음악인들이 작업물을 공유하고, 세션을 구하고,  
서로 협업할 수 있는 커뮤니티 기반 플랫폼으로 확장하는 것을 구상하고 있습니다.  

웹 서비스 안정화 이후에는 모바일 앱으로 확장하여  
음악 제작부터 편집, 공유까지 하나의 흐름 안에서 이루어지는 구조를 목표로 하고 있습니다.

## 시행착오와 설계 변화 과정 (5개월)

이 프로젝트는 초기 설계가 그대로 유지된 것이 아니라, 여러 차례 구조를 변경하며 개선되었습니다.

### 1. 이미지 기반 전체 처리 시도

초기에는 PDF를 이미지로 변환한 뒤, OMR로 음표와 가사, 코드를 모두 인식하는 방식을 시도했습니다.

문제점:
- Audiveris는 음표 인식은 가능했지만 가사 정확도가 낮았음
- 코드(chord symbol) 인식률이 불안정
- 이미지 왜곡과 해상도에 따라 결과 편차가 큼

이미지 기반 일괄 처리 방식은 안정성이 부족하다고 판단했습니다.

---

### 2. OMR + OCR 혼합 방식

음표는 Audiveris, 가사와 코드는 OCR로 별도 추출하여 병합하는 방식을 시도했습니다.

문제점:
- 워터마크 및 저작권 문구가 대량 혼입
- 코드 fragment 분해 문제 (예: C / m / 7)
- 줄바꿈 시 x좌표 단조 증가가 깨짐
- 최근접 매칭 방식의 구조적 한계

특히 줄이 바뀌는 순간 정렬이 붕괴되는 문제가 발생했습니다.

---

### 3. 최종 구조 선택

구조를 재설계했습니다.

- 음표는 Audiveris XML에서 직접 추출
- 가사 및 코드는 PDF 텍스트 레이어에서 추출 (pdfplumber)
- 단순 x좌표 매칭이 아닌 (measure, offset) 기준으로 재정의

좌표의 절대값이 아닌, 음악적 시간 개념으로 변환하는 방식으로 전환했습니다.

## 정렬 알고리즘 개선

문제:
- x좌표는 줄이 바뀌면 다시 초기화됨
- 단순 거리 기반 매칭은 정확도가 낮음

해결:
- page + system 단위 그룹화
- Hard Anchor 기준점 도입
- 좌/우 분리 정렬 방식 적용
- Dynamic Programming 기반 비용 함수 설계
- note capacity rule 적용

단순 최근접 매칭이 아니라 구조적 정렬 방식으로 개선했습니다.

## 구현 과정에서 해결한 주요 문제

- 줄바꿈 시 x좌표 단조 증가 깨짐 문제
- 코드 fragment 분해 및 병합 문제
- slash chord 병합 문제
- 가사 밀림 현상
- 워터마크 및 저작권 텍스트 혼입

좌표 간격과 문맥을 함께 고려하는 병합 로직을 구현하여 개선했습니다.
